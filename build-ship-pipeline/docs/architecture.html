Build & Ship Pipeline

Architecture reference — how all the pieces fit together

System Overview

A user types a feature request. Twelve LLM agents collaborate to produce working code, tests, Docker config, observability dashboards, and a security/quality audit — all within a hard cost ceiling. The whole pipeline runs in the background while a React UI streams progress in real time.

🤖

12 Specialised Agents

Planner, Coder, Docker, Observability, Tester, Debugger, Reviewer + 5 review specialists. Each has a fixed role and a default model.

🛡️

Budget Governance

Dynamic per-action token/cost caps tighten as budget depletes. A pre-flight guard blocks calls before they're made. Hard ceilings can never be exceeded.

🎛️

Skill System

Prompt injections append instructions to agent prompts. Agent toggles disable specific agents for a run. Both support per-session overrides on top of global defaults.

Pipeline State

All data flows through a single PipelineState TypedDict. Agents read from it and return a partial dict with only the fields they change. LangGraph merges the partial result with the current state between every node call.

Fields marked Accumulate use an Annotated[list, add] reducer — agents must return only the new items, not the full list.

Skill System

Skills let you customise pipeline behaviour without changing code. They are stored in the skills table and apply globally unless overridden per session.

prompt_injection

Appends the skill's prompt_addon text to the system prompt of each targeted agent. Used for tech-stack instructions, security checklists, style guides, etc.

Example: "Always write TypeScript. Never use any." targeting coder.

agent_toggle

Removes a specific agent from enabled_agents. The governed() decorator returns {} immediately — no LLM call, zero cost.

Example: skip-docker targeting docker.

Restriction: Cannot target planner or coder.

How context is computed at run start

1. Load all is_default=true skills from DB
2. Build effective[agent] = set(default_skill_ids) for every agent
3. Apply session skill_overrides: {"coder": {"add":["ts"], "remove":["eslint"]}, "*": {"add":["owasp"]}}
4. prompt_injection skills → join prompt_addon texts → skill_context[agent]
5. agent_toggle skills → disabled set → enabled_agents = ALL_AGENTS - disabled

Frontend Components

App.tsx — root

Auth gate, global skill state (skillOverrides), SkillsManager overlay toggle. Fetches /api/skills after auth.

Sidebar

Run history list (polls every 5s). Settings icon opens SkillsManager. New run button resets state.

ChatWindow

Feature textarea + ModelPicker + SessionSkills + Execute/Stop. Submits POST /api/runs with skill overrides.

StreamPanel

Real-time CoT: budget bars, per-agent step cards, modified files list, findings sorted by severity.

SessionSkills

Collapsible per-run override picker. Default skills shown as chips (click to disable). + button to add non-defaults. Per-agent accordion.

SkillsManager

Global CRUD panel. Create/edit/delete skills. Toggle is_default. System skills are protected (cannot delete, limited edits).

ApprovalGate

Modal that appears when approvalRequired=true arrives via WebSocket. Shows the plan JSON. Approve/Reject calls /api/runs/{id}/approve.

ModelPicker

Provider dropdown → context-sensitive fields. Anthropic/OpenAI/Groq only need model name. Ollama/Custom also need api_base. Any can supply api_key override.
